## 1. What This Engine Does — The Big Picture

The SynthData Engine is a tool that **generates training data for AI classifiers** using an AI (an LLM — Large Language Model) to write the examples.

A **classifier** is an AI model that reads a piece of text and puts it into a category. Examples:
- "Is this email spam or not spam?" → 2 categories
- "Which emotion does this sentence express: happy, sad, angry, or neutral?" → 4 categories
- "Which support department should handle this ticket: payments, security, billing, or tech?" → 4–5 categories

To train a classifier, you need **labeled examples** — thousands of texts where each text already has the correct category attached. Collecting these by hand is slow and expensive. This engine generates them automatically, at scale, using an LLM.

**What makes this hard:**
- If you just ask an LLM "give me 1000 examples of support tickets", it will give you 1000 variations of the same 3 templates. Real data is diverse — frustrated customers, calm customers, technical users, vague users, people who mention money, people who don't.
- If you need equal numbers of each category (balanced data), a naive LLM will ignore rare categories and only write common ones.
- If you need a realistic quality bar, many generated examples will be too generic, too short, or structurally correct but semantically wrong.

The six-stage pipeline solves each of these problems specifically.

## 2. Why Synthetic Data Exists — The Problem It Solves

### Real data problems

When building a classifier, you need labeled data. The traditional options are:

| Option | Problem |
|--------|---------|
| Collect real data manually | Slow, expensive, privacy issues |
| Hire annotators to label real data | Expensive, inconsistent |
| Use public datasets | Rarely matches your specific domain |
| Crowdsource | Quality inconsistent, hard to control edge cases |

**Synthetic data** — generated by an AI — solves most of these:
- Fast (minutes, not months)
- Cheap (LLM API costs vs human labor)
- Controllable (you can force specific categories, diversity levels, edge cases)
- Privacy-safe (no real user data, no PII)
- Extendable (need more data? Run again)

### What BYOK means

BYOK = **Bring Your Own Key**. This engine doesn't own or manage any AI service. You provide your own API credentials (AWS Bedrock or a local Ollama install). The engine just orchestrates calls to whatever AI you have access to.

## 3. The User Journey — From Config to Dataset

Here is exactly what a user does and what happens at each step:

### Step 1 — Write a config file

The user creates a YAML file (plain text with key:value pairs) that describes:
- What domain they're working in (fintech support, HR, emotion detection, etc.)
- What categories they need
- How many examples they want
- What quality bar to apply
- Which AI provider to use

Example snippet:
```yaml
project:
  name: FinTech Support Ticket Triage
  domain_brief: |
    Customer support tickets for a mobile payments app...

dataset:
  target_count: 50
  require_balanced: true

schema:
  fields:
    - name: text
      type: string
    - name: label
      type: enum
      values:
        - name: payment_dispute
          description: Customer reports money moved incorrectly
        - name: account_security
          description: Customer suspects fraud or unauthorized access
```

### Step 2 — Run the CLI

```bash
uv run synthdata configs/fintech_support_triage.yaml --target 50
```

That's it. The engine runs, prints progress, and saves a file.

### Step 3 — Get a JSONL file

The output is a file where each line is one training example:
```json
{"text": "I tried to pay my rent and the transaction failed three times. My bank says the money left my account but the landlord never received it. This is over $1,400 and I need it resolved today.", "label": "payment_dispute", "urgency": "high", "metadata": {"channel": "mobile_app", "mentions_amount": true, "is_boundary_case": false}}
{"text": "Someone logged into my account from Germany at 3am. I've never been to Germany. My linked debit card shows a charge I didn't make.", "label": "account_security", "urgency": "high", "metadata": {"channel": "web_portal", "mentions_amount": false, "is_boundary_case": false}}
```

### Step 4 — Use it to train a classifier

The JSONL file can be fed directly into any NLP training pipeline (HuggingFace, OpenAI fine-tuning, etc.).

## 4. The Six-Stage Pipeline — An Assembly Line for Data

Think of the pipeline like an assembly line in a factory. Raw material goes in one end (the config), finished product comes out the other end (the dataset). Each station does one specific job and passes its work to the next station.

```
Config
  │
  ▼
Stage 1: Axis Discovery ─────────── "What dimensions make examples different from each other?"
  │
  ▼
Stage 1.5: Logic Filter ────────── "Which combinations of those dimensions actually make sense?"
  │
  ▼
Stage 2: Allocation Plan ─────────  "How many examples should we generate per valid combination?"
  │
  ▼
Stage 3: Persona Pool ────────────  "Who are the fictional people who will 'write' these examples?"
  │
  ▼
Stage 4: Generation ──────────────  "Generate the actual examples, in parallel, with retries"
  │
  ▼
Stage 5: Judge ───────────────────  "Score each example 1–10 on quality, discard the bad ones"
  │
  ▼
Stage 6: Deduplication ───────────  "Remove near-duplicate examples"
  │
  ▼
Quality Gate ─────────────────────  "Filter by text length, report class distribution"
  │
  ▼
Output JSONL file
```

If after Stage 6 the engine is still short of the target count (because judge/dedup removed too many), it loops back and runs a **backfill** — regenerating only as many as needed to hit the target.

### 5.2 Entry Point — `cli.py`

**What it is:** The front door. When you type `uv run synthdata configs/...yaml`, this file runs first.

**What it does in detail:**

#### Lines 1–30: Imports
```python
import asyncio
import csv
import json
...
from rich.console import Console
from rich.table import Table
import typer
```

- `asyncio` — Python's library for running many things at once without blocking (like making 8 LLM calls simultaneously instead of one at a time)
- `typer` — turns Python functions into CLI commands (handles `--target 50 --output mydata.jsonl`)
- `rich` — makes the terminal output look nice (colored boxes, progress tables, highlighted errors)

#### The main function: `run()`

This is the function Typer calls when you run the CLI. It does:

1. **Load config** (`load_config(config_path)`) — reads your YAML file
2. **Apply CLI overrides** — if you passed `--target 50`, override the config's target_count
3. **Set up callbacks**:
   - `on_stage(stage, payload)` — called by the pipeline whenever a stage starts/finishes; the CLI logs this to the console
   - `on_progress(event)` — called every 10 samples during generation; prints "✓ 10 samples ok", "✓ 20 samples ok"...
4. **Run the pipeline** — `await run_job(cfg, ...)` — this is where all the work happens
5. **Write output** — calls `_write_output()` to save the dataset to disk
6. **Print summary table** — the final Rich table showing all stats and any warnings

#### Why `async` everywhere?

The keyword `async` appears throughout the codebase. Here's what it means for a beginner:

Imagine you need to ask 8 different friends for information. You could:
- **Synchronous (blocking):** Call friend 1, wait for them to reply, then call friend 2, wait, call friend 3... Total time = 8 × wait time.
- **Asynchronous (non-blocking):** Call all 8 friends at the same time, collect all replies when they come in. Total time = 1 × wait time.

LLM API calls take 2–5 seconds each. With `async` and `concurrency=8`, we make 8 calls simultaneously instead of 8 sequential calls. This is why target=50 takes ~60 seconds instead of ~7 minutes.

#### `_write_output()`

Takes the list of generated samples and writes them in the format the user requested:
- **JSONL** (default): one JSON object per line — best for machine reading
- **JSON**: a single JSON array — best for humans to read
- **CSV**: spreadsheet format — best for Excel/Pandas

#### `_print_summary()`

Generates the final summary table you see in the terminal. Reads `JobResult` (what the pipeline returned) and formats everything:
- How many samples were written
- Schema failure / refusal / error counts
- Judge pass/fail counts
- Distribution of enum fields (how many per category)
- Any CRITICAL or WARNING messages from the quality report

**Why this file exists as a separate layer:** The pipeline (`pipeline.py`) knows nothing about terminals, colors, or file formats. It just computes and returns data. The CLI layer handles all the "show the human what's happening" concerns. This separation makes the engine usable both from the CLI AND from Python code (you can call `run_job()` directly without a terminal).

### 5.3 Config System — `config.py`

**What it is:** The translator between your YAML file and Python objects the code can use.

**What Pydantic does:** Pydantic is a library that validates data. You tell it "this field must be a number between 0 and 1" and it will reject anything that doesn't fit, giving a clear error message. Every config model in this file inherits from Pydantic's `BaseModel`.

#### Key models (top to bottom):

**`Speakers`**
```python
class Speakers(BaseModel):
    in_scope: list[str] = []
    out_of_scope: list[str] = []
```
Describes who should and shouldn't be writing these texts. The LLM uses this to stay in character.

The `.as_prompt_block()` method formats this into text for the LLM prompt:
```
SPEAKERS (IN-SCOPE — personas must represent only these):
- Retail customer reporting a failed transaction
- Business owner with billing questions
...
SPEAKERS (OUT-OF-SCOPE — never generate a persona for these):
- Internal employees messaging each other
```

**`ProjectConfig`**
```python
class ProjectConfig(BaseModel):
    name: str
    domain_brief: str
    speakers: Speakers = Speakers()
```
The `domain_brief` is the most important field in the entire config. It grounds ALL stages:
- Stage 1 uses it to discover relevant axes
- Stage 1.5 uses it to judge plausibility
- Stage 3 uses it to generate appropriate personas
- Stage 4 injects it as context for every generation prompt

Think of `domain_brief` as "what world are we operating in?" The richer and more specific it is, the better the generated examples.

**`DatasetConfig`**
```python
class DatasetConfig(BaseModel):
    target_count: int
    brief: str
    require_balanced: bool = False
    diversity: Literal["standard", "high", "edge_cases"] = "standard"
    persona_pool_size: int = 40
    min_text_chars: int = 10
    max_text_chars: int = 2000
    axes: dict[str, list[str]] | None = None
```

- `require_balanced`: If `True`, the engine guarantees every enum class appears in the output. Critical for classifier training (you can't train a model that has never seen a class).
- `diversity`: Controls how "unusual" the generated examples should be. `"high"` asks the LLM to vary sentence structure, vocabulary, cultural context. `"edge_cases"` asks for ambiguous, borderline examples.
- `axes`: Advanced users can skip Stage 1 (axis discovery) and provide their own axes manually.

**`ProviderConfig`**
```python
class ProviderConfig(BaseModel):
    type: Literal["ollama", "bedrock"]
    concurrency: int = 8
    timeout_seconds: int = 120
    model: str | None = None
    host: str = "http://localhost:11434"
```

`concurrency` is how many simultaneous LLM calls to make. Set to 1 for Ollama (it can't handle parallel calls). Set to 8 for Bedrock (it handles concurrent API calls fine).

**`JudgeConfig`**
```python
class JudgeConfig(BaseModel):
    enabled: bool = True
    min_correctness: int = 7
    min_realism: int = 7
    min_distinctiveness: int = 6
```

Three quality dimensions, each scored 1–10:
- **Correctness** (7+): Does the example actually belong to the stated category? A "payment dispute" ticket that's really about account security would fail.
- **Realism** (7+): Does this read like something a real person wrote? No LLM-isms, no overly perfect grammar.
- **Distinctiveness** (6+): Is this different enough from the other examples? A very generic "my payment failed" when we already have 20 of those would fail.

**`AntiSeed`**
```python
class AntiSeed(BaseModel):
    text: str
    reason: str
```

Anti-seeds are explicit "don't generate anything like this" examples. The `text` is the bad example, `reason` explains why it's bad. Both are injected into every generation prompt.

**`JobConfig`** — the master config
```python
class JobConfig(BaseModel):
    project: ProjectConfig
    dataset: DatasetConfig
    schema: SchemaConfig
    seeds: list[dict] = []
    anti_seeds: list[AntiSeed] = []
    provider: ProviderConfig
    dedup: DedupConfig
    judge: JudgeConfig
    logic_filter: LogicFilterConfig
```

Note the `seeds` validation in the `__init__` (via `model_validator`): it runs `DatasetSchema.validate_row()` on every seed to make sure they match the schema. If your seed example has a label that isn't in the enum, it fails right here with a clear error before any LLM calls are made.

### 5.4 Schema System — `schema.py`

**What it is:** The blueprint for one training example. Every generated sample must match this blueprint exactly.

**Why this matters:** The LLM is given this schema and told to output JSON matching it. The engine validates every response against it. If the LLM made up an extra field, or used a label value that doesn't exist, it gets rejected and retried.

#### `FieldType`
```python
FieldType = Literal["string", "integer", "float", "boolean", "enum", "array", "object"]
```

Seven supported types. The last two (`array` and `object`) enable nesting — you can have a `metadata` object containing sub-fields, or a `spans` array containing objects with `start`, `end`, `label`.

#### `EnumValue`
```python
class EnumValue(BaseModel):
    name: str
    description: str = ""
```

Each allowed value for an enum field, plus an optional description. Descriptions are critical: they appear in the generation prompt as rules like:
```
"label" must equal "product_bug" — meaning: Customer reports app crashing, a feature not functioning, 
an error message appearing unexpectedly...
```
Without descriptions, the LLM often assigns the nearest-sounding category. With descriptions, it understands the semantic boundaries.

#### `SchemaField` — the self-referential recursive model
```python
class SchemaField(BaseModel):
    name: str = ""
    type: FieldType = "string"
    description: str = ""
    values: list[EnumValue] | list[str] | None = None   # for enum
    items: SchemaField | None = None                    # for array
    fields: list[SchemaField] | None = None             # for object
    min: float | None = None
    max: float | None = None
    required: bool = True
```

The `items` and `fields` fields reference `SchemaField` itself — this is a **recursive type definition**. An object field's `fields` is a list of more `SchemaField` objects, each of which can also be objects or arrays. This enables unlimited nesting depth.

```python
SchemaField.model_rebuild()   # Required for self-referential Pydantic models
```
This line at the end of the class definition tells Pydantic "this model references itself, finish building it now." Without it, Pydantic can't resolve the circular reference.

#### `DatasetSchema`
```python
class DatasetSchema(BaseModel):
    fields: list[SchemaField] = Field(..., max_length=20)
```

The top-level schema is just a list of fields (max 20 to prevent runaway configs).

**Key methods:**

**`.validate_row(row: dict) -> (bool, list[str])`**
Validates one generated sample against the schema. Returns `True` + empty errors, or `False` + list of error messages.

```python
ok, errs = schema.validate_row({"text": "hi", "label": "unknown_label", ...})
# ok = False
# errs = ["field 'label': 'unknown_label' not in ['payment_dispute', 'account_security', ...]"]
```

**`.json_example() -> dict`**
Returns a filled-in example of what valid JSON should look like. This is injected into every generation prompt so the LLM knows exactly what shape to produce:
```json
{
  "text": "<string>",
  "label": "payment_dispute",
  "urgency": "high",
  "metadata": {
    "channel": "mobile_app",
    "mentions_amount": true,
    "is_boundary_case": false
  }
}
```

**`.string_field_paths() -> list[str]`**
Returns dot-notation paths to all string-typed leaves. Used by dedup and quality to know which fields contain text that should be checked for similarity. For the fintech schema:
```python
["text"]  # metadata.channel is enum not string; metadata.* booleans aren't strings
```

**`extract_strings(row, fields) -> list[str]`**
Recursively extracts all string values from a row, including inside nested objects and arrays. Used by the quality gate to find the text content regardless of schema depth.

**`_check_type(field, val) -> str | None`**
Internal recursive function that validates a single field value. Returns `None` if valid, or an error message string if not. Handles all 7 types:
- `string`: checks `isinstance(val, str)` + optional length bounds
- `integer`: checks `isinstance(val, int)` + optional min/max
- `float`: checks `isinstance(val, (int, float))` + optional min/max
- `boolean`: checks `isinstance(val, bool)`
- `enum`: checks `val in allowed_names`
- `object`: recursively validates each sub-field in `field.fields`
- `array`: checks it's a list, then validates each item against `field.items`

### 5.5 Pipeline Orchestrator — `pipeline.py`

**What it is:** The conductor. It calls each stage in the right order, passes data between stages, handles errors and edge cases, writes checkpoints, and runs backfill loops.

**This is the biggest and most complex file (454 lines).** Let's walk through it section by section.

#### Imports and constants

```python
from .stages.plan import AllocationPlan, PlanRow, build_plan, SWEET_MIN
```

Everything the pipeline needs from the stage modules is imported at the top. This single file imports ALL the stages, while the stages only import from each other indirectly (through the pipeline's orchestration).

#### `JobResult` dataclass

```python
@dataclass
class JobResult:
    samples: list[dict]          # The final dataset
    axes: dict                   # What axes were discovered
    gen_stats: GenStats          # How many succeeded/failed during generation
    judge_stats: JudgeStats      # How many passed/failed the judge
    duplicates_removed: int      # How many near-dupes were removed
    elapsed_seconds: float       # Total wall-clock time
    quality_report: QualityReport # Distribution stats and warnings
    backfill_used: bool          # Was backfill triggered?
    ...
```

This is what `run_job()` returns — everything you might want to know about the run.

#### `run_job()` — the main function

The function signature:
```python
async def run_job(
    cfg: JobConfig,
    *,
    on_stage: Callable | None = None,        # Callback: called when a stage starts/finishes
    on_progress: Callable | None = None,     # Callback: called every N samples
    debug_dir: str | None = None,            # Path to write debug markdown files
    checkpoint_dir: str | None = None,       # Path for checkpoint files (resume support)
    dry_run: bool = False,                   # Just plan, don't actually generate
) -> JobResult:
```

**`*` in the function signature** means "everything after this must be passed by name, not by position." So you can't call `run_job(cfg, my_callback)` — you must call `run_job(cfg, on_stage=my_callback)`. This prevents mistakes.

**Callbacks** are functions you pass in. Instead of the pipeline printing things itself, it calls your function and lets you decide what to do with the event. The CLI passes in `on_stage=on_stage` so every event gets logged to the terminal. A web app might pass a different callback that sends events to a websocket.

#### Checkpoint loading (lines ~95–115)

```python
if checkpoint_dir:
    loaded_samples = _load_checkpoint(checkpoint_dir, cfg)
    if loaded_samples:
        samples = loaded_samples
        remaining = max(0, cfg.dataset.target_count - len(samples))
```

If you ran a previous job that was interrupted (power cut, API timeout, whatever) and passed `--checkpoint-dir`, the pipeline loads the samples already generated and only generates the remaining ones. This is especially valuable for large targets (10,000+ samples) where a full rerun would be expensive.

#### Stage 1 — Axes Discovery (lines ~118–135)

```python
if cfg.dataset.axes:
    # User provided axes manually in the config — skip discovery
    axes = cfg.dataset.axes
else:
    # Ask the LLM to discover axes
    axes = await discover_axes(cfg, provider, debug=dbg)
```

If the user already knows what dimensions they want, they can bypass the LLM call entirely. If not, the LLM figures it out.

#### Balanced mode injection (lines ~128–135)

```python
if cfg.dataset.require_balanced:
    for f in cfg.dataset_schema.fields:
        if f.type == "enum" and f.values and f.name not in axes:
            axes[f.name] = f.value_names()
```

This is the `require_balanced: true` mechanism. It forces every enum field from the schema (like `label` and `urgency`) to become a mandatory axis. This means every generated combination will have an explicit label value, so all classes MUST appear in the plan.

**Why this is necessary:** Without this, the LLM discovers axes like "tone" and "writing_style" but has no idea what labels should appear in what proportions. Common classes (like "account_security" — fraud is common in fintech datasets) would be over-represented while rarer classes (like "general_support" — how-to questions) would be under-represented or absent entirely.

#### Stage 1.5 — Logic Filter (lines ~137–163)

```python
max_useful = max(cfg.dataset.target_count * 3, 100)
if len(all_combos) > max_useful:
    combos_to_use = random.sample(all_combos, max_useful)
```

A critical pre-filter: the Cartesian product of axes can be enormous. With 5 labels × 3 urgency × 5 speakers × 3 tone = 225 combinations. With 8 axes that's 18,000+. Running the logic filter on all of them would itself take minutes and cost money.

The solution: randomly pre-sample down to `max(target_count * 3, 100)` combinations before even running the filter. For target=50, this means we pass at most 150 combinations to the filter. The filter runs on these 150 and keeps the plausible ones.

#### Stage 2 — Plan (lines ~165–200)

```python
balanced_fields = (
    [f.name for f in cfg.dataset_schema.fields if f.type == "enum" and f.values]
    if cfg.dataset.require_balanced else None
)
plan = build_plan(valid, cfg.dataset.target_count, balanced_by=balanced_fields, debug=dbg)
```

`balanced_by` tells the planner "when downsampling combinations to fit the budget, distribute slots equally across these field values so no class is starved."

**Balanced coverage guarantee** (lines ~169–185):
```python
if cfg.dataset.require_balanced:
    for f in cfg.dataset_schema.fields:
        if f.type != "enum" or not f.values:
            continue
        covered = {r.combination.get(f.name) for r in plan.rows}
        for ev in f.values:
            if ev.name not in covered:
                plan.rows.append(PlanRow(
                    combination_id=uuid4().hex[:12],
                    combination={f.name: ev.name},  # minimal combo — only pin required field
                    target=SWEET_MIN * 3,            # 15 samples to absorb judge losses
                ))
```

This is the safety net. Even after stratified sampling, if some class is still missing from the plan (due to bad luck or a very skewed valid set), a fallback row is added. The fallback uses a **minimal combination** — only the required field is pinned. This gives the LLM maximum freedom to generate text that naturally fits that class, avoiding axis conflicts that would cause the judge to fail everything.

#### Stage 4 — Generation (lines ~205–240)

```python
samples = flatten_samples(plan)
```

After generation runs, all samples from all plan rows are collected into a single flat list.

#### Stage 5 — Judge (lines ~241–260)

```python
if cfg.judge.enabled:
    samples, judge_stats = await judge_samples(samples, cfg, provider=provider, debug=dbg)
```

Optional. If enabled, each sample is scored by the LLM. Only samples meeting minimum thresholds survive.

#### Stage 6 — Dedup (lines ~265–278)

```python
text_fields = pick_text_fields(cfg.dataset_schema)
samples, removed = dedupe(samples, text_fields=text_fields, threshold=cfg.dedup.threshold)
```

Near-duplicate removal. `pick_text_fields()` finds all string-leaf fields in the schema (recursively), so dedup works on nested schemas too.

#### Backfill loop (lines ~280–350)

```python
for _pass in range(1, _MAX_BACKFILL_PASSES + 1):
    if len(samples) >= cfg.dataset.target_count:
        break

    gap = cfg.dataset.target_count - len(samples)
    overshoot_factor = 1.4 + (_pass - 1) * 0.3  # 1.4×, 1.7×, 2.0×, 2.3×, 2.6×
    backfill_plan = build_plan(valid, int(gap * overshoot_factor))
    ...
```

If after judge + dedup we're short of the target, we generate more. The overshoot factor escalates with each pass (1.4×, 1.7×, ...) because if we needed a backfill, the previous overshoot wasn't enough. Escalating ensures convergence. After 5 passes, if still short, the engine gives up and returns what it has.

### 5.6 Quality System — `quality.py`

**What it is:** The inspector at the end of the assembly line. Checks text lengths, counts class distributions, fires warnings.

#### `apply_quality_gate(samples, cfg_dataset, schema) -> (kept, removed_count)`

Filters out samples where the text is too short (`< min_text_chars`) or too long (`> max_text_chars`). Uses `extract_strings()` from `schema.py` to find the text regardless of nesting.

Why filter by length?
- Too short (< 10 chars): Usually a generation failure — the LLM output a stub or placeholder
- Too long (> 2000 chars): Usually indicates the LLM rambled or included JSON metadata it shouldn't

#### `compute_report(samples, schema, length_filtered=0) -> QualityReport`

Analyses the final dataset:
- For each enum field: counts how many samples have each value
- For each numeric/float field: computes min, max, mean
- For each string field: computes character length min, max, mean

**Imbalance detection:**
```python
expected_per_class = total_samples / len(values)
threshold = expected_per_class * 0.6   # 60% of equal share

if count == 0:
    warnings.append(CRITICAL: "class X has 0 samples")
elif count < threshold:
    warnings.append(WARNING: "class X has only N samples, expected ~M")
```

A CRITICAL fires when a class has zero samples — you literally cannot train a classifier on a class it has never seen. A WARNING fires when a class has less than 60% of its fair share. Both are shown in the terminal summary and also emitted as Python warnings.

#### `FieldDistribution` dataclass

```python
@dataclass
class FieldDistribution:
    field_name: str
    counts: dict[str, int]      # enum value → count
    min_val: float | None       # for numeric fields
    max_val: float | None
    mean_val: float | None
    is_char_count: bool = False  # True = this is character length, not a real value
```

The `is_char_count` flag distinguishes between "the confidence field has mean=0.87" (a real value) vs "the text field has mean=342 chars" (a length metric). The CLI formats them differently: `:.3f` for real values, `:.0f` for character counts.

### 5.8 LLM Providers

#### `providers/base.py` — The Contract

```python
class LLMResponse:
    outcome: str           # "success", "invalid_json", "refusal", "timeout", "throttled", "error"
    raw_text: str          # Exactly what the LLM returned (before any parsing)
    parsed: dict | None    # The parsed JSON (or None if parsing failed)
    error: str | None      # Error message if outcome != "success"
    attempts: int          # How many retries were needed
```

Every provider returns this same `LLMResponse` object. The pipeline code only knows about `LLMResponse` — it doesn't care whether it came from Bedrock or Ollama. This is the **abstraction layer**.

```python
class LLMProvider(Protocol):
    model: str
    async def generate_json(self, system, user, ...) -> LLMResponse: ...
    async def generate_text(self, system, user, ...) -> LLMResponse: ...
```

A `Protocol` in Python is like a contract. Any class that has a `model` attribute and these two async methods is automatically a valid `LLMProvider`. You could add a Claude provider, a Gemini provider, an OpenAI provider — as long as they implement these two methods, the pipeline will work without any changes.

#### `providers/bedrock.py` — AWS Bedrock

**What it does:** Wraps the `boto3` AWS SDK to talk to AWS Bedrock's language model API.

**BedrockCredentials:**
```python
@classmethod
def from_env(cls) -> BedrockCredentials:
    return cls(
        access_key=os.getenv("AWS_ACCESS_KEY_ID"),
        secret_key=os.getenv("AWS_SECRET_ACCESS_KEY"),
        region=os.getenv("AWS_REGION", "us-east-1"),
        model_id=os.getenv("AWS_MODEL_ID", "amazon.nova-lite-v1:0"),
    )
```

Reads credentials from environment variables (set in your `.env` file). Never hardcoded in source code — that would be a security risk.

**Throttling backoff:**
```python
for attempt in range(max_retries + 1):
    try:
        response = client.converse(...)
        break
    except ThrottlingException:
        delay = min(2 ** attempt * base_delay + jitter, 120)
        await asyncio.sleep(delay)
```

AWS rate-limits API calls. When throttled, the engine waits and retries with exponential backoff (wait 1s, then 2s, then 4s... up to 120s). The `jitter` (small random amount) prevents multiple concurrent calls from all retrying at the same exact moment (the "thundering herd" problem).

**Why `bedrock-runtime.converse` instead of model-specific APIs?** Bedrock's Converse API is a unified interface that works the same way for Claude, Llama, Nova, Titan, etc. Using it means switching models is just changing the `model_id` string — no code changes needed.

#### `providers/ollama.py` — Local Ollama

**What it does:** Wraps the Ollama Python library to talk to a locally-running Ollama server.

**JSON repair logic:**
```python
def _repair_json(raw_text: str) -> dict | None:
    # Try 1: Extract from markdown code fence (```json ... ```)
    match = re.search(r"```(?:json)?\s*([\[{].*?)\s*```", raw_text, re.DOTALL)
    if match:
        return json.loads(match.group(1))
    
    # Try 2: Find outermost {} or []
    start = raw_text.find('{') or raw_text.find('[')
    end = raw_text.rfind('}') or raw_text.rfind(']')
    return json.loads(raw_text[start:end+1])
```

Local models (gemma, llama, mistral) frequently return JSON wrapped in markdown fences, or with extra explanation text before/after it. The repair logic strips all that away to get the actual JSON.

Bedrock models are generally better at following "return JSON only" instructions, but the repair logic runs there too as a fallback.

**Why Ollama is slower:** Ollama runs the model on your local GPU/CPU. Each token generates sequentially, and there's only one instance running. AWS Bedrock runs on dedicated server infrastructure with multiple instances. Additionally, Ollama must serialize requests because the model can only process one at a time (`NUM_PARALLEL=1`).

### 5.9 Stage 1 — Axis Discovery: `discover.py`

**What it is:** A creative brainstormer. Given the domain, what dimensions make examples different from each other?

**Why we need this:** If you just generate 50 "payment dispute" tickets without thinking about what should vary, you'll get 50 variations of "my payment didn't go through." Real training data needs variation in:
- Who's writing (frustrated customer vs calm business owner)
- What information they include (specific amounts vs vague complaints)
- Their emotional tone (outraged vs matter-of-fact)
- How technically detailed they are

These dimensions are the **axes**. Each axis has 2–5 discrete values.

**The prompt:**
```python
_DISCOVER_USER = """List the 3–6 most important VARIATION AXES for generating diverse training examples in this domain.

DOMAIN BRIEF:
{domain_brief}

...

For each axis, provide:
- axis_name: a short identifier (snake_case)
- values: 2–5 distinct values this axis can take

Return JSON:
{"axes": {"axis_name": ["val1", "val2", ...], ...}}
"""
```

The LLM responds with something like:
```json
{
  "axes": {
    "speaker": ["retail_customer", "business_owner", "account_security_customer", "app_user"],
    "urgency_signal": ["urgent_language", "question", "statement", "complaint", "request"],
    "complexity": ["simple", "moderate", "detailed"],
    "frustration_level": ["calm", "slightly_annoyed", "frustrated", "very_frustrated"]
  }
}
```

**Reconciliation:**
```python
for ax_name, ax_values in axes.items():
    schema_field = next((f for f in cfg.dataset_schema.fields if f.name == ax_name), None)
    if schema_field and schema_field.type == "enum" and schema_field.values:
        axes[ax_name] = schema_field.value_names()  # Force to schema values
```

If the LLM discovered an axis called "label" (matching a schema enum field), its values are overridden to match the schema exactly. Without this, the LLM might use "payment_issue" when the schema says "payment_dispute" — a mismatch that would fail validation.

**Temperature 0.3:** This stage uses a low temperature (less random) because we want a coherent, thoughtful set of axes, not a wild brainstorm.

### 5.10 Stage 1.5 — Logic Filter: `logic_filter.py`

**What it is:** A plausibility checker. Of the (potentially thousands of) combinations of axis values, which ones actually make sense?

**`cartesian(axes) -> list[Combination]`**

Given:
```python
axes = {
    "speaker": ["retail_customer", "business_owner"],
    "urgency": ["high", "medium", "low"],
    "label": ["payment_dispute", "account_security"]
}
```

Returns all 2 × 3 × 2 = 12 combinations:
```python
[
  {"speaker": "retail_customer", "urgency": "high", "label": "payment_dispute"},
  {"speaker": "retail_customer", "urgency": "high", "label": "account_security"},
  {"speaker": "retail_customer", "urgency": "medium", "label": "payment_dispute"},
  ...
]
```

**`filter_combinations(combos, ...) -> list[Combination]`**

Batches the combinations and sends them to the LLM:
```
Combinations (numbered):
0. {"speaker": "account_security_customer", "label": "billing_inquiry", "urgency": "high"}
1. {"speaker": "retail_customer", "label": "payment_dispute", "urgency": "low"}
2. {"speaker": "general_support_customer", "label": "product_bug", "urgency": "medium"}
...

Mark a combination as IMPLAUSIBLE only if it's contradictory, absurd, or
socially/emotionally incoherent...

Return JSON: {"keep": [<list of indices to KEEP>]}
```

**Example removals:**
- `speaker=retail_customer, label=general_support, urgency=high` might be flagged: "general_support is inherently low urgency — high urgency + general_support is incoherent"
- `speaker=account_security_customer, label=billing_inquiry, urgency=low` might survive: "an account security customer with a low urgency billing question is plausible"

**Fail-open design:**
```python
try:
    result = parse(response)
except:
    return all_combinations  # On any parse error, keep everything
```

If the logic filter LLM call fails or returns garbage, we keep all combinations rather than risking losing valid ones. A missed invalid combination means one slightly weird example gets generated — not the end of the world. Accidentally removing valid combinations means missing entire classes — much worse.

### 5.11 Stage 2 — Allocation Plan: `plan.py`

**What it is:** A resource allocator. Given N valid combinations and a target of M total samples, how many should we generate per combination?

**The math:**

```
adjusted_target = target_count × 1.1    (10% overshoot to cover judge/dedup losses)
per_combo = adjusted_target ÷ n_combos

Example: target=50, n_combos=11
adjusted = 55
per = 55 ÷ 11 = 5 per combo
```

**SWEET_MIN and SWEET_MAX constants:**
- `SWEET_MIN = 5`: Never generate fewer than 5 from one combination. With fewer, you lose the diversity that one combination can offer (different personas, different seeds).
- `SWEET_MAX = 20`: Never generate more than 20 from one combination. Beyond this, you're just getting variants of the same scenario — diminishing returns.

**What happens when there are too many combinations:**
```python
# For target=50, adjusted=55, n_combos=150:
per = 55 ÷ 150 = 0  → max(1, 0) = 1  (too few)
# Need per ≥ SWEET_MIN = 5
# Downsample: keep = 55 ÷ 5 = 11
combinations = stratified_sample(combinations, balanced_by, 11)
```

The stratified sampler runs when `balanced_by` is set:

**`_stratified_sample(combinations, balanced_by, keep, rng)`**

Uses a **greedy coverage algorithm**:
1. For every candidate combination, compute a score: sum of `1/(coverage+1)` for each balanced field value it covers
2. Pick the highest-scoring combination (the one that covers the most under-represented values)
3. Update coverage counts
4. Repeat until `keep` combinations are selected

Example trace with 5 labels and keep=11:
- First pick: covering `label=payment_dispute` and `urgency=high` (both coverage=0, score=2.0)
- Second pick: covering `label=account_security` and `urgency=medium` (score=2.0)
- Third pick: covering `label=product_bug` and `urgency=low` (score=2.0)
- Fourth pick: covering `label=billing_inquiry` and `urgency=high` (urgency=high coverage=1, score=1.5)
- Fifth pick: covering `label=general_support` and `urgency=medium` (score=1.5)
- Sixth through eleventh: filling remaining slots, now biased toward less-covered values

Result: approximately 2–3 combinations per label value, with urgency also distributed.

**`AllocationPlan` dataclass:**
```python
@dataclass
class AllocationPlan:
    target_count: int
    rows: list[PlanRow]

    @property
    def total_target(self) -> int:
        return sum(r.target for r in self.rows)

    @property
    def undershoot_risk(self) -> bool:
        return self.combination_capped and self.total_target < self.target_count
```

`undershoot_risk` is True when even after giving every combination the maximum allowed samples, we still can't reach the target. This happens with very small combination spaces (e.g., only 2 valid combinations, target=100, MAX_PER_COMBO=20 → max achievable = 40). The pipeline warns the user if this happens.

### 5.12 Stage 3 — Prompt Composer: `compose.py`

**What it is:** The scriptwriter. Takes all the context (domain, schema, combination, persona, seeds) and assembles the exact text sent to the LLM for each generation.

#### Persona Pool

**Why personas?** Without personas, all generated examples tend to sound the same — the LLM's default "neutral assistant" voice. With personas, each example is written by a different imagined person:
- "A 45-year-old small business owner, stressed about cash flow, writes in terse direct sentences"
- "A tech-savvy millennial, calm but methodical, provides detailed error messages and version numbers"
- "A retired woman, unfamiliar with technology, frustrated and using caps lock for emphasis"

`build_persona_pool()` generates 40 such personas (by default) by asking the LLM:
```
Generate {n} distinct personas that could plausibly AUTHOR messages in this domain.
Each persona is a short description (12-25 words) covering: role, context, writing style.

HARD CONSTRAINTS:
- Every persona must represent an IN-SCOPE speaker.
- Never invent a persona for an OUT-OF-SCOPE speaker type.
```

#### `compose_prompt()` — assembling the generation prompt

This function takes:
- `combination`: the specific axis values for this call (e.g., `{label: "product_bug", urgency: "medium"}`)
- `schema`: what fields the output must have
- `seeds`: 1-2 reference examples (for tone/style guidance)
- `anti_seeds`: examples to explicitly avoid
- `personas`: the pool, one randomly chosen
- `diversity`: standard | high | edge_cases

And produces the full (system, user) pair sent to the LLM.

**System prompt:**
```
You generate ONE realistic training example for an NLP dataset.
Output strict JSON matching the requested schema. No prose, no explanation.
```

Short, authoritative, no room for the LLM to add commentary.

**User prompt structure:**
```
Generate ONE example matching this schema:
{json_example}

CONTEXT FOR THIS SPECIFIC EXAMPLE:
- label: product_bug
- urgency: medium
- speaker: app_user
- emotional_tone: frustrated
- Speaker persona: Tech-savvy mobile user in their 30s, writes concisely, uses technical terms
- Unique seed: a3f29b11

REFERENCE EXAMPLES (for tone and shape only — DO NOT COPY):
{"text": "Every time I add a new payee the app freezes...", "label": "product_bug", ...}

ANTI-EXAMPLES (DO NOT generate anything resembling these):
- "My payment failed and I'm very angry!" — reason: Too vague, no useful information

DIVERSITY DIRECTIVE: Generate an UNUSUAL, VARIED example. Vary sentence length, vocabulary...

HARD RULES:
- "text" must be a string
- "label" must equal "product_bug" — meaning: Customer reports app crashing, a feature not functioning...
- "urgency" must equal "medium" — meaning: Functional issue affecting usage but no immediate risk
- "metadata" must be an object matching the structure shown above
```

**`_build_rules(schema, combination)` — generating the HARD RULES section:**

For each field in the schema:
- If the field is **pinned** by the combination (appears in the combination dict): `"field" must equal "value" — meaning: description`
- If the field is **free** (not in the combination): `"field" must be one of: * "value1" — description * "value2" — description`
- If the field is **numeric with bounds**: `"field" must be between min and max`
- If the field is **object** or **array**: `"field" must be an object/array matching the structure shown above`

This is the most important part of the prompt. The HARD RULES section is what forces the LLM to generate correctly-labeled, schema-compliant examples.

### 5.13 Stage 4 — Generator: `generate.py`

**What it is:** The factory floor. Makes the actual LLM calls, handles all the ways they can fail, and collects the results.

#### Concurrency model

```python
sem = asyncio.Semaphore(cfg.provider.concurrency)  # default: 8

async def _one_call(row: PlanRow) -> None:
    async with sem:  # Only 8 calls can run at once
        ...

tasks = [asyncio.create_task(_one_call(row)) for row in plan.rows for _ in range(row.target)]
await asyncio.gather(*tasks)
```

All `row.target × n_rows` tasks are created at once. The semaphore ensures at most 8 run simultaneously. This is how 55 LLM calls happen in ~10 seconds instead of ~110 seconds.

#### Error handling hierarchy

```python
# 1. LLM didn't return valid JSON at all (network error, timeout, etc.)
if resp.outcome != "success":
    if resp.outcome == "refusal":
        stats.refusals += 1
        row.status = "failed"
        return  # Skip this combination
    # Corrective retry: tell the LLM what went wrong
    correction = user + f"\n\nYour previous response failed: {resp.error}. Return valid JSON."
    resp = await provider.generate_json(system, correction, max_retries=1)
    if still failed:
        stats.errors += 1; return

# 2. JSON parsed but doesn't match the schema
ok, errs = schema.validate_row(resp.parsed)
if not ok:
    # Schema-corrective retry
    correction = user + f"\n\nSchema validation failed: {errs}. Fix ONLY the flagged fields."
    resp2 = await provider.generate_json(system, correction, max_retries=1)
    if still failed:
        stats.schema_failed += 1; return

# 3. Success — but enforce pinned fields
for key, pinned_val in row.combination.items():
    if key in schema_field_names and key in resp.parsed:
        resp.parsed[key] = pinned_val  # Force the label to match the plan
```

**The pinned field enforcement (the fix from this session):**

When `require_balanced: true` is set, the plan pins `label=product_bug` for a particular combination. The prompt HARD RULES also say `"label" must equal "product_bug"`. But LLMs can drift — especially if the text they naturally generate reads more like an account security issue. Without enforcement, the sample would be labeled `account_security` even though it was planned as `product_bug`.

The fix: after schema validation, force-set any field that appears in both the combination dict AND the schema. This ensures the label always matches what was planned. The text is still generated naturally (only the label field is overridden), and since the generation prompt was already contextualized for `product_bug`, the text and label should match.

#### `flatten_samples(plan) -> list[dict]`

After all tasks complete, each `PlanRow` has a `.samples` list. This function concatenates them all:
```python
return [sample for row in plan.rows for sample in row.samples]
```

### 5.14 Stage 5 — Judge: `judge.py`

**What it is:** The quality control inspector. Reads each generated sample and scores it on three dimensions, filtering out the bad ones.

#### Why judge at all?

Without a judge, LLMs sometimes generate:
- Examples that are technically valid JSON but semantically wrong ("payment dispute" ticket that's clearly about a software bug)
- Examples that are too generic to be useful training data
- Examples that are copies of the seed examples
- Examples that sound like an AI wrote them, not a human

The judge catches all of these.

#### The judge prompt

```python
_JUDGE_USER = """You are evaluating training data quality. Score this example on 3 dimensions:

DOMAIN: {domain_brief}
SCHEMA: {schema_hint}
EXAMPLE TO JUDGE: {sample_json}

SCORE EACH DIMENSION 1-10:
- correctness: Does it actually belong to the stated label?
- realism: Does it read like real human writing? (Not AI-ish, not templated)
- distinctiveness: Is it meaningfully different from generic examples in this class?

Think step by step. Then provide JSON: {{"correctness": N, "realism": N, "distinctiveness": N}}
"""
```

The "think step by step" instruction is important. It forces the judge LLM to reason about the example before scoring, rather than just guessing a number. This is called **Chain-of-Thought (CoT) prompting** — a well-established technique that significantly improves LLM judgment accuracy.

#### Fail-open design

```python
try:
    scores = _extract_scores(resp.raw_text)
    passes = (
        scores["correctness"] >= jcfg.min_correctness and
        scores["realism"] >= jcfg.min_realism and
        scores["distinctiveness"] >= jcfg.min_distinctiveness
    )
except:
    passes = True   # On any parse error, let the sample through
    stats.unparseable += 1
```

If the judge LLM itself fails (timeout, bad JSON response), the sample PASSES. This is the fail-open principle: we'd rather let a slightly-below-bar sample through than accidentally discard good data because the judge was having a bad day. The consequence of a false pass is one mediocre example in the dataset. The consequence of a false reject is losing data (potentially the last example of a rare class).

### 5.15 Stage 6 — Deduplication: `dedup.py`

**What it is:** The duplicate remover. After generation, some examples are near-copies of each other (same scenario, slightly different phrasing). This removes them.

#### MinHash LSH — how it works

**The problem:** With 1000 examples, you'd need to compare every pair: 1000 × 999 ÷ 2 = ~500,000 comparisons. That's too slow.

**MinHash:** Converts text into a compact "fingerprint" (a set of hash numbers). Two texts with similar fingerprints are probably similar. Computing fingerprints is fast.

**LSH (Locality Sensitive Hashing):** Groups similar fingerprints into "buckets" so you only compare texts within the same bucket. Near-duplicates end up in the same bucket; completely different texts don't.

Together, MinHash LSH makes deduplication **near-linear time** — even 100,000 examples can be deduplicated in seconds.

#### How similarity is measured

**Jaccard similarity:**
```
Text A tokens: {payment, failed, account, bank}
Text B tokens: {payment, error, account, bank}

Intersection: {payment, account, bank}  → 3
Union: {payment, failed, error, account, bank}  → 5

Jaccard = 3/5 = 0.6 (60% similar)
```

The default threshold is **0.85 (85%)** — two texts must share 85% of their token content to be considered duplicates. This is conservative: it removes obvious near-copies but preserves paraphrases.

Example of what this catches:
- "My payment failed yesterday" vs "my payment failed yesterday" (case variant) → 100% similar → removed
- "My payment failed" vs "My transfer failed" → ~70% similar → kept (different token "transfer" vs "payment")

#### `pick_text_fields(schema) -> list[str]`

```python
def pick_text_fields(schema) -> list[str]:
    strings = schema.string_field_paths()
    if strings:
        return strings
    return [f.name for f in schema.fields]
```

Uses `string_field_paths()` to find all string leaf fields in the schema (including nested ones like `"metadata.source"` or `"spans[].text"`). Only string fields contain human-readable text worth deduplicating on. Enum fields (like `"label"`) would collapse all same-label examples into one bucket — we don't want that.

#### `_resolve_path(row, path) -> list[str]`

```python
def _resolve_path(row: dict, path: str) -> list[str]:
    parts = path.replace("[]", "[*]").split(".")
    current: list = [row]
    for part in parts:
        if part.endswith("[*]"):
            # Array expansion: collect all items from this array
            key = part[:-3]
            next_level = []
            for node in current:
                if isinstance(node, dict):
                    val = node.get(key, [])
                    if isinstance(val, list):
                        next_level.extend(val)
            current = next_level
        else:
            # Normal key lookup
            next_level = []
            for node in current:
                if isinstance(node, dict) and part in node:
                    next_level.append(node[part])
            current = next_level
    return [str(v) for v in current if isinstance(v, str)]
```

This handles paths like:
- `"text"` → `row["text"]`
- `"metadata.source"` → `row["metadata"]["source"]`
- `"spans[].text"` → `[item["text"] for item in row["spans"]]`

It's a mini path-query engine that supports nested objects and array expansion. Without this, dedup would only work on flat schemas.

## 6. Config Files — Domain Examples

### `configs/emotion_detection_bedrock.yaml`

**Domain:** 4-class emotion classifier: happy, sad, angry, out_of_scope

The `out_of_scope` class is special — it's a catch-all for messages that aren't expressing one of the three emotions. Without it, a classifier trained only on happy/sad/angry would have no way to handle neutral messages or off-topic ones.

**Key design choices in this config:**
- Seeds: 5 examples (one per class plus `out_of_scope`)
- Anti-seeds: messages that are ambiguous, automated, or obviously fake
- `require_balanced: true`: critical here — `out_of_scope` is easy to miss without it
- `min_correctness: 7, min_realism: 8, min_distinctiveness: 6`: elevated realism bar because emotion classification depends heavily on authentic human phrasing

### `configs/hr_intent_bedrock.yaml`

**Domain:** HR intent classifier for a job description management system

Four classes:
- `create_jd` — create a new job posting
- `refine_jd` — improve an existing posting
- `delete_jd` — remove a posting
- `hide_jd` — temporarily hide without deleting

These are easy to confuse: "update the JD" could be `refine_jd` or `delete_jd + create_jd`. The seed examples and anti-seeds focus on disambiguating these boundary cases.

### `configs/fintech_support_triage.yaml`

**Domain:** 5-class support ticket router for a fintech company

This is the most complex config because:
- 5 classes with significant overlap (a payment that failed due to an app crash touches both `payment_dispute` AND `product_bug`)
- Nested metadata object (`channel`, `mentions_amount`, `is_boundary_case`)
- Secondary label (`urgency`: high/medium/low)
- `min_realism: 8` — the highest realism bar of any config, because support ticket classifiers need to handle frustrated real-world language, not clean examples

The `is_boundary_case: true` flag in metadata marks examples that could plausibly belong to two categories — these are explicitly valuable training signal. A classifier needs to learn where the boundaries are.

## 7. Tests — How We Know It Works

**92 tests across 9 test files.** All run in under 1 second (no actual LLM calls — tests use mock providers).

### Philosophy

The tests use **mock providers** — fake LLMs that return pre-written responses. This means:
- Tests run instantly (no network latency)
- Tests are deterministic (same input = same output, always)
- Tests can be run offline (no credentials needed)

The tradeoff: mock tests can't catch issues with specific LLM behaviors (the real LLMs might interpret prompts differently). Those are caught by the integration/validation runs (like the fintech validation we did).

### Test structure by file

**`test_schema.py`** — Tests every data type, every validation rule:
- Valid rows pass
- Invalid enum values fail with the right error message
- Float out of range fails
- Missing required fields fail
- Extra undeclared fields fail
- Recursive validation: nested object fails correctly, array item fails correctly
- `json_example()` produces the right nested shape
- `string_field_paths()` returns correct dot-notation paths
- `extract_strings()` pulls strings from nested structures

**`test_plan.py`** — Tests the allocation algorithm:
- With enough combos: each gets the right per-combo target
- With too many combos: stratified sampling used correctly
- SWEET_MAX respected
- Undershoot risk correctly flagged
- Total target = adjusted_target (with overshoot)

**`test_quality.py`** — Tests the quality reports:
- CRITICAL fires when count = 0
- WARNING fires below 60% threshold
- No warnings when balanced
- Float field stats computed correctly
- String char counts vs float values formatted differently

**`test_dedup.py`** — Tests near-duplicate detection:
- Case variants detected and removed
- Paraphrases preserved
- Empty input handled
- Nested path dedup works

## 8. Bugs Found and Fixed During Production Validation

During the FinTech Support Triage production validation run (April 2026), three real bugs were discovered and fixed. Understanding them helps you understand the system's edge cases.

### Bug 1: Plan Downsampling Drops Entire Classes

**What happened:** The engine ran with `require_balanced: true` on a 5-class problem. The result showed `product_bug: 0` — an entire class had zero samples.

**Root cause (in `plan.py`):**
```python
# With target=50, n_valid_combos=150:
per = 55 // 150 = 1   # Too few per combo
# Need per >= SWEET_MIN=5, so downsample:
keep = 55 // 5 = 11   # Keep only 11 combinations
combinations = rng.sample(combinations, 11)  # RANDOM sample!
```

The random sample of 11 from 150 happened to select no `product_bug` combinations. With `require_balanced: true`, this should never happen.

**Fix:** Replaced the random sample with a stratified greedy sampler (`_stratified_sample`) that distributes slots proportionally across all required field values.

**Fix location:** `stages/plan.py`, `_stratified_sample()` function and `build_plan()` signature.

### Bug 2: LLM Ignores Pinned Label — "Label Drift"

**What happened:** The plan correctly included a `product_bug` row (after the stratified sampler fix). The LLM was given the prompt with HARD RULE `"label" must equal "product_bug"`. But the LLM generated text about account security and labeled the output `account_security`. Schema validation passed (it's a valid enum value). The sample appeared in the dataset as `account_security`, not `product_bug`.

**Root cause:** The schema validator only checks that `"account_security"` is a valid enum value. It doesn't check that it matches the pinned value in the combination.

**Fix (in `generate.py`):**
```python
# After schema validation passes, enforce the combination's pinned fields
schema_field_names = {f.name for f in schema.fields}
for key, pinned_val in row.combination.items():
    if key in schema_field_names and key in resp.parsed:
        resp.parsed[key] = pinned_val   # Force label to match the plan
```

The combination is ground truth — it defines what class this example SHOULD be. If the LLM drifted, we correct it.

**Fix location:** `stages/generate.py`, after schema validation block.

### Bug 3: Fallback Combination Creates Axis Conflicts

**What happened (after Bug 1 fix):** A fallback row was added for `product_bug` using a random combination from `valid` like `{speaker: "account_security_customer", ..., label: "product_bug"}`. The LLM generated text, but the judge failed ALL 5 samples from this row. `product_bug` still had 0 samples in the final output.

**Root cause:** The combination `speaker=account_security_customer + label=product_bug` creates a conflict. The LLM generated account-security-flavored text (because the speaker is an account security customer), then the label enforcement (Bug 2 fix) forced the label to `product_bug`. The judge correctly identified this as unrealistic: "this text is about unauthorized access, not an app bug."

**Fix (in `pipeline.py`):**
```python
# Use MINIMAL combination — only pin the required field
plan.rows.append(PlanRow(
    combination_id=uuid4().hex[:12],
    combination={f.name: ev.name},   # Just {label: product_bug}
    target=SWEET_MIN * 3,            # 15 samples to absorb judge losses
))
```

With a minimal combination, the LLM has maximum freedom. It generates text naturally appropriate for `product_bug` (because that's all that's pinned). No axis conflicts, no judge failures.

**Fix location:** `pipeline.py`, balanced coverage guarantee block.

## 10. Design Decisions and Trade-offs

### Why 6 separate stages?

Separation of concerns. Each stage has one job. If the judge is causing too many rejections, you can tune only the judge config. If generation is slow, you can increase concurrency only in the provider config. You can disable stages you don't need (logic filter, judge) without touching other stages.

### Why overshoot by 10%?

The pipeline generates 10% more samples than needed (`adjusted_target = target × 1.1`). This means if the judge rejects 5–8% of samples (typical) and dedup removes 1–2%, you still usually hit the target without a backfill pass. The overshoot costs a few extra LLM calls but saves the latency of a full backfill round.

### Why MinHash instead of embedding-based semantic dedup?

**Embedding-based dedup** (using a vector similarity model) catches semantic duplicates — "payment failed" and "transaction didn't complete" would be flagged as similar even though they share no tokens.

**MinHash** only catches surface-level duplicates — token overlap. "Payment failed" and "Transaction failed" would not be flagged.

The choice was intentional: **paraphrases are valuable diversity**. If two examples say the same thing in different words, both belong in the training data — they help the classifier learn the same concept expressed differently. We only want to remove cases where the LLM essentially output the same sentence twice (case variants, punctuation changes, word order swaps).

### Why Chain-of-Thought in the judge?

Making the judge reason step-by-step before scoring significantly improves score reliability. Without CoT, the LLM tends to give high scores to anything that looks like it belongs to the right class. With CoT, it actually thinks about "does this read like a real human? Is there something off about it?"

### Why fail-open for the judge?

If the judge LLM times out or returns bad JSON, the sample passes automatically. The alternative — failing closed (reject on any error) — would mean a flaky network connection causes you to lose data. Since the judge's job is quality control (not security), the failure mode of "letting through a slightly bad sample" is much preferable to "losing data on network errors."

### Why not use OpenAI / Claude instead of Bedrock?

This is a BYOK engine. Users bring their own credentials. The goal is not to force a specific provider. Bedrock is the default because:
- AWS credentials are common in enterprise environments
- Bedrock provides access to multiple model families through one API
- The Converse API gives a consistent interface across Claude, Llama, Nova, Mistral, etc.

Adding other providers (OpenAI, Anthropic directly, Google Vertex) is straightforward — implement the 2-method `LLMProvider` protocol.

## 12. Glossary

**Async / Asynchronous:** Running multiple operations concurrently without blocking. In this engine, it means making 8 LLM calls at the same time instead of waiting for each one.

**Axis:** One dimension of variation. "Writing tone" is an axis with values like "formal", "casual", "frustrated". A combination is one specific value from each axis.

**Backfill:** A retry pass that generates extra samples when the main run comes up short due to judge/dedup losses.

**BYOK (Bring Your Own Key):** The engine doesn't host any AI service. You connect it to your own AWS Bedrock or Ollama credentials.

**Cartesian Product:** All possible combinations of axis values. If there are 3 labels × 2 urgency levels = 6 combinations.

**Chain-of-Thought (CoT):** Prompting a LLM to "think step by step" before giving an answer. Dramatically improves reasoning quality.

**Combination:** One row of the Cartesian product — a specific set of axis values. E.g., `{label: product_bug, urgency: medium, tone: frustrated}`.

**Concurrency:** How many LLM calls happen simultaneously. Default 8 for Bedrock.

**Fail-open:** When a check fails (timeout, parse error), treat it as passing rather than failing. Used in judge and logic filter.

**Jaccard Similarity:** How similar two sets of tokens are: `|A ∩ B| / |A ∪ B|`. Used by dedup.

**Judge:** The quality evaluation LLM that scores each generated example 1–10 on correctness, realism, and distinctiveness.

**JSONL:** JSON Lines format. Each line of the file is a separate, complete JSON object. Standard format for ML training data.

**LSH (Locality Sensitive Hashing):** Groups similar items into buckets so you only compare items in the same bucket. Makes dedup near-linear time.

**MinHash:** A technique to create compact text fingerprints that can be quickly compared for similarity.

**Overshoot:** Generating 10% more examples than needed upfront to compensate for judge/dedup losses.

**Persona:** A fictional character description (role, writing style, context) injected into generation prompts to create diversity in voice and tone.

**Pydantic:** A Python library for data validation. Config models defined with Pydantic auto-validate all values against declared types and constraints.

**Require Balanced:** A config flag that guarantees every enum class appears in the output. Critical for classifier training.

**Schema:** The blueprint for one example — which fields it has, what types they are, what values are allowed.

**Seed:** An example provided by the user that shows the LLM the desired tone, style, and format. The LLM uses seeds for reference, not copying.

**Anti-seed:** A "bad" example provided by the user that shows the LLM what NOT to generate.

**Semaphore:** A concurrency control that limits how many tasks run at once. Used to limit LLM API concurrency to the configured value.

**SWEET_MIN / SWEET_MAX:** The sweet spot range for samples per combination (5–20). Ensures variety within each combination without generating redundant examples.

**Throttling / Backoff:** When an API rate-limits your requests, the engine waits and retries with exponentially increasing delays.
