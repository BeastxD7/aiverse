# v0.2.0 — Enterprise Hardening + 100K Scale (April 2026)

## Summary

Major reliability and scale update. The pipeline now handles datasets up to 100K samples reliably, with guaranteed class coverage, checkpoint/resume for interrupted runs, multi-pass backfill, and comprehensive quality reporting.

---

## Breaking changes

- CLI syntax changed: `uv run synthdata <config.yaml>` (no `run` subcommand)
- Output filename default changed from `out.jsonl` to `out_<timestamp>.jsonl`
- `judge.min_distinctiveness` default lowered from 7 → 6 (better pass rate on creative domains)
- Imbalance warning threshold raised from 50% → 60% of expected fair share (catches subtler imbalances)

---

## New features

### 100K scale: dynamic allocation planner

`SWEET_MAX=20` per-combination cap was a hard wall that made large targets impossible with typical axis sets (30 combos × 20 = 600, never reaching 100K). The planner now uses a dynamic effective ceiling:

```
effective_max = max(SWEET_MAX, ceil(adjusted_target / n_combos))
effective_max = min(effective_max, MAX_PER_COMBO=5000)
```

With 30 combos and target=100K, each combination handles ~3,334 samples. With 500 combos, ~200 samples each. `SWEET_MAX=20` still applies when combos are plentiful enough to hit the target at that cap.

### Multi-pass backfill (up to 5 passes)

Previously a single backfill pass ran at 1.4× the gap. With a strict judge in a narrow domain, one pass often left the pipeline still short. Now the pipeline loops up to 5 passes with escalating overshoot (1.4×, 1.7×, 2.0×, 2.3×, 2.6×), converging in 2–3 passes in practice.

### Checkpoint / resume

Large runs can now be interrupted and resumed without losing progress.

```bash
uv run synthdata configs/emotion.yaml --target 10000 --checkpoint-dir .checkpoints
# Ctrl+C at sample 4,231 → re-run same command to resume from sample 4,232
```

The checkpoint key is a content hash of domain-determining config fields (project, schema, seeds, provider). Changes to `--target`, `concurrency`, or judge thresholds do NOT invalidate the checkpoint.

### Balanced class coverage: `require_balanced`

```yaml
dataset:
  require_balanced: true
```

Injects every enum label field as a generation axis post-discovery, forcing every combination to explicitly target one class. Eliminates the zero-sample class problem for minority/subtle labels (e.g. `out_of_scope`, `neutral`).

### Diversity directives: `dataset.diversity`

```yaml
dataset:
  diversity: high   # standard | high | edge_cases
```

Injected into generation prompts to steer sample variety without changing axes.

### Pinned axes: `dataset.axes`

```yaml
dataset:
  axes:
    tone: [formal, casual]
    channel: [email, chat]
```

Skips Stage 1 entirely. Fully reproducible runs — same axes always produce the same checkpoint hash.

### Text quality gate + quality report

```yaml
dataset:
  min_text_chars: 50
  max_text_chars: 500
```

Post-dedup filter removes samples outside the character range. The run summary now shows per-field distributions and imbalance warnings:

```
│ label  │ happy: 10  sad: 9  angry: 10  out_of_scope: 11 │
│ WARNING│ field 'label' value 'happy' has only 5 samples... │
```

`CRITICAL` fires when a class has 0 samples. `WARNING` fires when a class is below 60% of its expected fair share.

### Dry-run mode

```bash
uv run synthdata configs/emotion.yaml --dry-run
```

Runs Stage 1 + Stage 1.5 only, prints the plan table (combinations, estimated LLM calls, estimated tokens), and exits. No generation cost.

### Bedrock throttle detection + exponential backoff

Detects `ThrottlingException`, `TooManyRequestsException`, `RequestLimitExceeded`, `ServiceUnavailableException`, `ModelNotReadyException` from AWS and applies `min(30 × attempt, 120)s + jitter` backoff before retrying. Non-throttle errors surface immediately.

### Per-call temperature control

`temperature: float = 0.8` is now a first-class parameter on both provider methods. Axis discovery uses `temperature=0.3` for deterministic axes. Generation and judging use `0.8` for variety.

### Judge model override

```yaml
provider:
  type: bedrock
  model: google.gemma-3-27b-it-qat-q8-0:2:2
  judge_model: anthropic.claude-3-5-haiku-20241022-v1:0
```

Use a stronger model for judging and a cheaper one for generation.

### Judge concurrency

```yaml
judge:
  concurrency: 8   # independent of provider.concurrency
```

## Pipeline changes

| Stage | Change |
|---|---|
| Stage 1 — Discover | Uses `temperature=0.3`; respects pinned `dataset.axes` |
| Stage 2 — Plan | Dynamic `effective_max`; `MAX_PER_COMBO=5000` hard cap |
| Stage 4 — Generate | `on_sample` checkpoint callback; scale-aware progress logging (every 1% of target) |
| Stage 5 — Judge | Configurable concurrency; `unparseable` counter; judge model override |
| Post-pipeline | Multi-pass backfill loop; text quality gate; quality report with imbalance warnings |

## Quality report

`quality.py` — new module. Runs after final dedup:
- Per-field distributions (enum counts, string char lengths, float value stats)
- Imbalance warnings at 60% threshold (`WARNING`) and 0-sample threshold (`CRITICAL`)
- `filtered_by_length` count for text quality gate

## Performance at scale

| Provider | Model | target=40 | target=1000 | target=100K (estimated) |
|---|---|---|---|---|
| AWS Bedrock | gemma-3-27b-it | ~52s | ~15 min | ~15–20 hours* |
| Ollama (M-series) | gemma4:e4b | ~90 min | — | Not recommended |

*With checkpoint/resume. Actual time depends on AWS throttling. Run will pause and back off during throttle windows automatically.

---

## Known limitations

- 100K runs require `--checkpoint-dir` — a crash without it restarts from Stage 1
- Debug mode (`--debug-dir`) at 100K produces very large log files; not recommended above target=1000
- Class balance is not mathematically guaranteed — judge filtering can still produce mild imbalance (~±20%); the quality report warns when this occurs
- Ollama is not recommended above target=100 due to serialized request model
