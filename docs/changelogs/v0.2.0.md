## New features

### 100K scale: dynamic allocation planner

`SWEET_MAX=20` per-combination cap was a hard wall that made large targets impossible with typical axis sets (30 combos Ã— 20 = 600, never reaching 100K). The planner now uses a dynamic effective ceiling:

```
effective_max = max(SWEET_MAX, ceil(adjusted_target / n_combos))
effective_max = min(effective_max, MAX_PER_COMBO=5000)
```

With 30 combos and target=100K, each combination handles ~3,334 samples. With 500 combos, ~200 samples each. `SWEET_MAX=20` still applies when combos are plentiful enough to hit the target at that cap.

### Multi-pass backfill (up to 5 passes)

Previously a single backfill pass ran at 1.4Ã— the gap. With a strict judge in a narrow domain, one pass often left the pipeline still short. Now the pipeline loops up to 5 passes with escalating overshoot (1.4Ã—, 1.7Ã—, 2.0Ã—, 2.3Ã—, 2.6Ã—), converging in 2â€“3 passes in practice.

### Checkpoint / resume

Large runs can now be interrupted and resumed without losing progress.

```bash
uv run synthdata configs/emotion.yaml --target 10000 --checkpoint-dir .checkpoints
# Ctrl+C at sample 4,231 â†’ re-run same command to resume from sample 4,232
```

The checkpoint key is a content hash of domain-determining config fields (project, schema, seeds, provider). Changes to `--target`, `concurrency`, or judge thresholds do NOT invalidate the checkpoint.

### Balanced class coverage: `require_balanced`

```yaml
dataset:
  require_balanced: true
```

Injects every enum label field as a generation axis post-discovery, forcing every combination to explicitly target one class. Eliminates the zero-sample class problem for minority/subtle labels (e.g. `out_of_scope`, `neutral`).

### Diversity directives: `dataset.diversity`

```yaml
dataset:
  diversity: high   # standard | high | edge_cases
```

Injected into generation prompts to steer sample variety without changing axes.

### Pinned axes: `dataset.axes`

```yaml
dataset:
  axes:
    tone: [formal, casual]
    channel: [email, chat]
```

Skips Stage 1 entirely. Fully reproducible runs â€” same axes always produce the same checkpoint hash.

### Text quality gate + quality report

```yaml
dataset:
  min_text_chars: 50
  max_text_chars: 500
```

Post-dedup filter removes samples outside the character range. The run summary now shows per-field distributions and imbalance warnings:

```
â”‚ label  â”‚ happy: 10  sad: 9  angry: 10  out_of_scope: 11 â”‚
â”‚ WARNINGâ”‚ field 'label' value 'happy' has only 5 samples... â”‚
```

`CRITICAL` fires when a class has 0 samples. `WARNING` fires when a class is below 60% of its expected fair share.

### Dry-run mode

```bash
uv run synthdata configs/emotion.yaml --dry-run
```

Runs Stage 1 + Stage 1.5 only, prints the plan table (combinations, estimated LLM calls, estimated tokens), and exits. No generation cost.

### Bedrock throttle detection + exponential backoff

Detects `ThrottlingException`, `TooManyRequestsException`, `RequestLimitExceeded`, `ServiceUnavailableException`, `ModelNotReadyException` from AWS and applies `min(30 Ã— attempt, 120)s + jitter` backoff before retrying. Non-throttle errors surface immediately.

### Per-call temperature control

`temperature: float = 0.8` is now a first-class parameter on both provider methods. Axis discovery uses `temperature=0.3` for deterministic axes. Generation and judging use `0.8` for variety.

### Judge model override

```yaml
provider:
  type: bedrock
  model: google.gemma-3-27b-it-qat-q8-0:2:2
  judge_model: anthropic.claude-3-5-haiku-20241022-v1:0
```

Use a stronger model for judging and a cheaper one for generation.

### Judge concurrency

```yaml
judge:
  concurrency: 8   # independent of provider.concurrency
```

## require_balanced bug fixes (production validation â€” April 2026)

Three bugs found and fixed during a production validation run (fintech support triage, target=50, 5-class):

**Bug 1 â€” Plan downsampling drops entire label classes**
`build_plan` downsamples combinations using `rng.sample()` (purely random) when there are too many combinations for the target. The random draw could select zero combinations for a given label value.
Fix: added `_stratified_sample()` â€” a greedy coverage-maximising sampler that scores combinations by `sum(1/(coverage+1))` across all balanced fields, guaranteeing every enum value is represented in the allocation plan.

**Bug 2 â€” LLM label drift ignores pinned label**
The generation prompt hard-pins a label (e.g. `label=product_bug`) but the LLM occasionally outputs a different valid enum value. Schema validation passes (the value is valid) so the mislabelled sample enters the dataset silently.
Fix: after schema validation in `generate.py`, force-set all schema fields that appear in the combination: `resp.parsed[key] = pinned_val`.

**Bug 3 â€” Fallback combinations cause 100% judge failure**
The post-plan fallback rows (inserted for uncovered enum values) used full combinations from `valid` â€” e.g. `{speaker: account_security_customer, label: general_support}`. The axis conflict caused the judge to correctly reject all generated samples.
Fix: fallback rows now use minimal combinations `{field_name: value}` with `target=SWEET_MINÃ—3`. The model generates naturally appropriate text with no axis conflicts.
